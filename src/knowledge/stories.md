# Behavioral Stories

## Leadership — JiffyJobs Team Lead
On JiffyJobs I led a 5-person team through 5 sprints over a semester. Early on I made the mistake of trying to control too much — reviewing every PR, jumping into areas that weren't mine, not fully trusting my teammates. We missed early milestones and I was burning out. The turning point was when I stepped back and actually delegated: let Vinay own the chat module, Hrutika own the map and task features, Purvang own testing. When I played to everyone's strengths instead of trying to be everywhere, we shipped faster and built a better product. That shift in mindset — leadership means trusting your team — is something I carry into every project now.

## Failure & Learning — CI/CD on JiffyJobs
First time setting up a CI/CD pipeline. I assumed it would be straightforward — it wasn't. The GitHub Actions workflow kept failing: TypeScript build errors, Prisma client generation timing, DB migrations running before the build finished, environment variables missing in CI. I spent nights debugging. What I learned: read the docs thoroughly before starting, understand the execution order, and don't assume local = CI. By the end I had a working pipeline that ran tests, built both frontend and backend, and auto-deployed to Vercel and Render on every push. First-time struggle became a skill I'm proud of.

## Going Above and Beyond — StemAlly Accessibility Research
When I joined Prof. Palani's lab, my job was to build iOS features. But I went deeper — I studied VoiceOver extensively, read WCAG guidelines cover to cover, tested the app myself blindfolded to understand the actual user experience. I reached out to learn about the Carroll Center for the Blind where testing was conducted. Most developers build features. I tried to genuinely understand what it feels like to be blind and use a touch interface. That perspective shift made my code better.

## Learning Fast Under Pressure — .NET/C# for AlertFlow
AlertFlow required a .NET 10 GraphQL API — I had never used .NET or C# before. I had a week to build it. I went deep: watched tutorials, read Microsoft docs, built small prototypes, then built the real thing. The multi-stage Docker build, Hot Chocolate GraphQL setup, Redis caching — all new to me. I shipped it in a week. The lesson: strong fundamentals (understanding APIs, databases, architecture patterns) transfer across languages. I can pick up new tech quickly when I understand the underlying concepts.

## Teamwork — Cross-functional at SharkNinja
At SharkNinja I worked with Product, QA, and Backend engineers on a consumer app used by real customers. I learned to write code that other people depend on — production discipline, clear PR descriptions, not breaking things in main. Debugging Bluetooth connectivity issues required working closely with hardware engineers — a very different collaboration than pure software teams.

## Conflict/Disagreement — Handling Ambiguity
I always try to find the positive step forward even in difficult situations. When there's a disagreement on technical approach, I prefer to prototype both options quickly and let the results speak rather than argue in theory. I've found that most technical disagreements dissolve when you actually test the approaches.

## Handling Failure — SharkNinja
SharkNinja was a challenging internship. It was a production environment with high standards, and there were aspects of my performance that weren't where I wanted them to be. I took it as a learning experience — production codebases are different from school projects. The discipline required, the communication standards, the speed expected — it all raised my bar. I came out of it more aware of what production-level engineering looks like and more motivated to meet that standard.
